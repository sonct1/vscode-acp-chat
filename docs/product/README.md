# Product Documentation

Use this area for durable product context: what the service must achieve, who uses it, business flows, requirements, acceptance criteria, non-goals, dependencies, risks, and open questions.

Create product docs only when product scope is not obvious from code, API contracts, or feature docs.

Typical files:

- `service-prd.md` — service-level product requirements.
- `<feature>-prd.md` — feature-level product requirements.
- `business-flows.md` — core business flows and user/system actors.

Do not put API payloads, database schema, technical designs, implementation plans, task lists, deployment steps, or troubleshooting runbooks here.
